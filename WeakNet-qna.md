# WEAK_NET 项目面试八股问题与参考答案

本文档面向简历项目面试使用，围绕 WEAK_NET 本地弱网监控与诊断系统展开。内容包括系统整体设计、Linux 网络、C++ 常驻服务、netlink、ICMP、TCP 重传率、Wi-Fi RSSI、eBPF、D-Bus、Web Dashboard、SQLite、AI/RAG、工程落地和排障优化。

## 项目难点

异构网络指标的时间对齐问题： RTT、TCP 重传、RSSI、流量统计等指标来自不同采集路径，采样周期和更新时机不一致，容易出现某一时刻的诊断结果混用了不同时间窗口的数据。项目中通过统一状态缓存、周期采样和日志时间戳解析，将多源指标汇总到同一分析窗口内，保证诊断结果具备一致性。

弱网场景下的误判控制： 单一指标无法准确反映真实网络质量，例如 RTT 抖动可能来自公网目标波动，RSSI 对有线网卡无意义，低流量时 TCP 重传率也容易失真。因此需要对无效值、缺失值和异常波动进行过滤，并结合多指标权重进行综合判断，降低误报和漏报。

底层采集能力与运行环境差异的兼容： eBPF、RSSI、系统网络计数器等能力对内核版本、权限、网卡类型和运行环境依赖较强，在虚拟机、有线网卡或权限不足时可能无法完整采集。项目需要设计降级路径，例如 eBPF 不可用时回退到 /sys/class/net 和 /proc/net/snmp，保证系统在不同 Linux 环境下仍能输出可用诊断结果。

## 一、系统整体理解

### 1. 这个项目解决什么问题？

这个项目解决的是 Linux 终端、网关、边缘设备上的本地弱网感知和网络诊断问题。

传统排查方式通常依赖人工执行 `ping`、`ip route`、`ifconfig`、`ss`、`tcpdump` 等命令，效率低且缺少持续监控能力。本项目通过 C++ 常驻服务持续采集当前上网网卡、默认网关、RTT、TCP 重传率、Wi-Fi RSSI、流量速率、网络质量事件等指标，再通过 D-Bus、C API、Web Dashboard 和 AI 诊断报告对上层提供可视化和自动分析能力。

核心价值是把“人工临时排查”变成“端侧持续监控 + 结构化指标 + 自动诊断”。

### 2. 项目的整体架构是什么？

![result (2)](D:\Download\QuarkDownload\论文\result (2).svg)

整体分为五层：

1. **采集层**：C++ 常驻服务 `weaknet-dbus-server`，负责采集网卡、路由、RTT、RSSI、TCP 重传率、eBPF 流量等数据。
2. **通信层**：D-Bus 服务 `com.example.WeakNet`，对外暴露方法调用和事件信号。
3. **客户端层**：`libweaknet.so` 动态库和命令行测试工具，方便业务进程调用。
4. **可视化层**：Python Web Dashboard，读取日志、SQLite、系统计数器，展示实时指标、趋势、事件和诊断结果。
5. **AI 分析层**：基于日志/结构化指标和网络知识库，通过 RAG 或规则分析生成网络问题诊断建议。

数据流大致是：

```text
Linux 网络状态 /proc /sys /netlink /eBPF
        -> C++ weaknet-dbus-server
        -> D-Bus / 日志 / 事件
        -> Dashboard SQLite
        -> Web 页面 / AI 诊断报告
```

### 3. 为什么这个项目适合部署在 Linux 终端、网关和边缘设备？

因为这类设备通常具备以下特点：

- 长时间运行，需要持续监控。
- 网络问题常见，但现场人工排查成本高。
- 多数运行 Linux，具备 `/proc`、`/sys`、netlink、eBPF 等能力。
- 可能没有完整桌面环境，因此本地 Web Dashboard 比桌面 App 更适合。
- 边缘设备常需要判断问题发生在本机、网关、上游链路还是无线信号侧。

该项目以本地 agent 形式运行，不依赖中心化平台即可完成基础诊断。

### 4. 项目中哪些指标最关键？

核心指标包括：

- **当前上网网卡**：判断实际流量出口。
- **默认网关**：判断本机到局域网出口是否正常。
- **RTT**：判断链路延迟。
- **TCP 重传率/丢包率**：判断传输层质量。
- **RSSI**：判断 Wi-Fi 信号强弱。
- **Traffic/Bps/PPS**：判断流量负载和突发流量。
- **Top Flow**：定位具体连接或进程。
- **网络质量事件**：用于上层应用感知网络状态变化。

### 5. 项目中 C++ 服务和 Python Dashboard 的边界是什么？

C++ 服务负责底层采集和事件产生，靠近系统和内核；Python Dashboard 负责展示、缓存、兜底采集和诊断报告。

这样设计有几个原因：

- C++ 更适合做长期运行的底层网络采集服务。
- D-Bus/C API 便于被其他业务进程集成。
- Python 开发 Dashboard、SQLite 和规则诊断效率更高。
- 当 eBPF 或日志采集失败时，Dashboard 可以用 `/sys`、`/proc` 做兜底。

### 6. 这个项目最大的技术难点是什么？

主要难点有：

- 如何准确判断当前真正上网的网卡。
- 如何把 Linux 内核网络统计转化为可理解的指标。
- 如何处理 Wi-Fi、有线、虚拟机、eBPF 权限不足等复杂环境。
- 如何避免无效值如 `RTT=-1`、`RSSI=-1000` 影响诊断。
- 如何设计可落地的降级路径，保证系统在不同 Linux 环境下都能输出有价值的信息。

### 7. 项目为什么需要降级方案？

因为实际部署环境差异很大。

例如：

- eBPF 需要内核支持和权限，不一定能 attach 成功。
- VMware 虚拟网卡没有 Wi-Fi RSSI。
- 某些环境 ICMP 权限或绑定接口 ping 可能失败。
- 日志可能不完整或服务重启。

所以项目设计了多级数据来源：

- 首选 C++ 服务采集结果。
- 其次解析日志。
- 再其次用 Dashboard 直接读取 `/sys/class/net`、`/proc/net/snmp`、`/proc/net/route`。

这样可以提升系统可用性。

## 二、Linux 网络与网卡管理

### 8. 如何判断一个网卡是否具备上网能力？

可以从多个维度判断：

1. 网卡是否存在。
2. 网卡是否 UP/RUNNING。
3. 是否有 IPv4/IPv6 地址。
4. 是否参与默认路由。
5. 是否能通过该网卡访问探测目标。

项目中通过网络接口信息、路由表和当前默认路由综合判断具备上网能力的网卡，并维护网卡列表。

### 9. 为什么不能只看网卡 UP 状态？

因为 UP 只表示网卡被启用，不代表它能上网。

例如：

- 网卡 UP 但没有 IP。
- 网卡 UP 但没有默认路由。
- 虚拟网卡 UP 但不是公网出口。
- Docker、lo、桥接网卡也可能 UP，但不是实际上网接口。

所以必须结合 IP 地址、默认路由和实际探测结果。

### 10. 默认路由和当前上网网卡是什么关系？

默认路由决定了当目标地址没有更具体路由匹配时，流量从哪个网关和网卡出去。

例如：

```bash
default via 192.168.45.2 dev ens33
```

这说明当前默认流量出口是 `ens33`，默认网关是 `192.168.45.2`。

项目通过默认路由识别当前上网网卡，并在 `NetInfo` 中打 `usingNow` 标记。

### 11. 如何从 `/proc/net/route` 解析默认网关？

`/proc/net/route` 中默认路由的 Destination 为 `00000000`，Gateway 是小端十六进制。

例如 Gateway 字段 `022DA8C0` 需要按小端转换为 `192.168.45.2`。

Dashboard 中使用该方式识别默认网关，避免依赖 `ip route` 命令。

### 12. netlink 在这个项目里有什么作用？

netlink 是用户态和内核网络子系统通信的重要机制。

在项目里可以用于：

- 获取网卡列表。
- 监听网卡新增/删除。
- 获取网卡状态变化。
- 获取路由信息。
- 获取 TCP socket/统计信息。

相比解析命令输出，netlink 更结构化、更适合程序化处理。

### 13. netlink 相比执行 shell 命令有什么优势？

优势包括：

- 不依赖外部命令。
- 返回结构化数据。
- 性能更好。
- 可以监听事件。
- 更适合长期运行的服务。

执行 `ip route`、`ifconfig` 适合人工排查，但不适合作为稳定服务的数据源。

### 14. 如何处理网卡 DOWN 或移除？

服务端周期性采集当前接口列表，并与上一次列表做 diff：

- 新增的接口加入管理。
- 消失或不满足条件的接口移除。
- 状态变化时更新对应 `NetInfo`。
- 触发 `InterfaceChanged` 事件。

这样上层可以实时感知网卡变化。

### 15. 为什么要给当前上网网卡打标？

一台设备可能有多张网卡，如：

- 有线网卡
- Wi-Fi
- 蜂窝网络
- Docker/虚拟网卡

但实际公网流量通常只走默认路由对应的某一张网卡。打标后：

- 日志能快速看出当前上网方式。
- 质量评分可以聚焦当前链路。
- Dashboard 可以默认展示活跃网卡。
- 上层应用可以对网络切换做响应。

## 三、RTT 与 ICMP Ping

### 16. RTT 是什么？

RTT 是 Round Trip Time，即一次网络请求从发送到收到响应的往返时间。

在弱网诊断中，RTT 反映网络延迟。RTT 越高，用户感知到的卡顿越明显。

### 17. 为什么项目要封装 ICMP Ping？

封装 ICMP Ping 可以避免依赖外部 `ping` 命令，并且可以：

- 指定网卡发包。
- 控制超时时间。
- 统一日志格式。
- 与网卡管理逻辑集成。
- 周期性更新每张网卡的 RTT。

### 18. 为什么要对 DNS/公网目标做 RTT 探测？

默认网关 RTT 只能说明本机到局域网出口是否正常，不能说明公网链路是否正常。

公网目标如 `223.5.5.5` 可以反映本机到外部网络的连通性和延迟。

对比两者可以定位问题：

- 网关 RTT 高：本机到网关有问题。
- 网关 RTT 正常，公网 RTT 高：上游链路可能拥塞。
- 网关和公网都不可达：本机网络或路由可能异常。

### 19. 为什么系统 ping 命令能通，但服务端 RTT 可能是 -1？

可能原因包括：

- 服务端绑定指定网卡发包失败。
- ICMP socket 权限不足。
- DNS 解析或超时逻辑不同。
- 服务端 timeout 设置较短。
- 虚拟机环境下接口绑定行为与普通 ping 不同。

因此 Dashboard 增加了自身 ping 兜底探测。

### 20. RTT 多少算正常？

一般经验：

- 到默认网关：通常小于 1 到 5 ms。
- 到公网 DNS：几十 ms 属于正常。
- 超过 80 到 100 ms 可能影响交互体验。
- 超过 200 ms 通常认为较差。

但阈值应根据业务场景调整。

### 21. 为什么不能只用 RTT 判断网络质量？

因为 RTT 只反映延迟，不反映：

- 丢包
- TCP 重传
- Wi-Fi 信号
- 带宽占用
- 突发流量
- DNS 失败
- 应用服务端异常

所以项目综合 RTT、TCP 重传率、RSSI、Traffic 等指标判断。

## 四、Wi-Fi RSSI

### 22. RSSI 是什么？

RSSI 是 Received Signal Strength Indicator，即接收信号强度，单位通常是 dBm。

数值越接近 0，信号越强。例如：

- `-40 dBm` 很强
- `-60 dBm` 中等
- `-75 dBm` 较弱
- 小于 `-80 dBm` 通常较差

### 23. 项目如何获取 Wi-Fi RSSI？

项目在检测到当前网卡是 Wi-Fi 网卡时，通过 socket 与 `wpa_supplicant` 控制接口通信，获取指定 Wi-Fi 网卡的 RSSI。

### 24. 为什么 VMware 下 RSSI 是 N/A？

VMware 中常见网卡如 `ens33` 是虚拟有线网卡，不是真实 Wi-Fi 网卡。

RSSI 只对无线网卡有意义，因此有线网卡环境下显示 `N/A` 是正常现象。

### 25. 为什么不能对所有网卡都采集 RSSI？

RSSI 是无线信号指标。以太网、虚拟网卡、Docker 网桥等没有无线信号概念。

如果对非 Wi-Fi 网卡强行赋值，会导致质量评分误判。

### 26. RSSI 在网络诊断中有什么价值？

它可以判断弱网是否由无线侧引起。

例如：

- RTT 高 + RSSI 弱：可能是 Wi-Fi 信号问题。
- RTT 高 + RSSI 正常：可能是上游链路或拥塞问题。
- TCP 重传高 + RSSI 弱：可能存在无线干扰或距离过远。

## 五、TCP 重传率与丢包率

### 27. TCP 重传率和网络丢包是什么关系？

TCP 重传通常是丢包、乱序、超时或拥塞导致的结果。

实际系统中很难直接得到“真实丢包率”，所以常用 TCP 重传率作为传输层质量指标。

### 28. 项目如何计算 TCP 重传率？

项目通过内核网络统计数据获取：

- TCP 发送段数量
- TCP 重传段数量

在一个采样周期内计算差值：

```text
重传率 = retrans_delta / out_segments_delta * 100%
```

Dashboard 兜底方案中读取 `/proc/net/snmp` 的 `Tcp OutSegs` 和 `Tcp RetransSegs`。

### 29. 为什么短窗口 TCP 重传率可能出现 200% 这种异常值？

短窗口差值计算时，如果分母很小，例如 `OutSegs` 只有 1 或 2，而 `RetransSegs` 增加较多，就可能出现很大的百分比。

这属于采样窗口过短导致的尖峰，不一定代表真实链路质量。

项目后来通过以下方式修正：

- 设置最小发送段阈值。
- 忽略超过合理范围的尖峰。
- 诊断时使用平均值和 P95，而不是直接使用 max。

### 30. 为什么诊断中不用 max 判断 TCP 重传异常？

因为 max 对偶发尖峰非常敏感。

弱网判断更适合使用：

- 平均值
- P95
- 连续异常次数
- 样本数量

这样可以避免单个异常点误报。

### 31. TCP Loss 为 0.00% 代表什么？

通常代表当前时间窗口内没有观察到 TCP 重传，链路在传输层上比较稳定。

它不是“没有数据”，而是一个有效的好结果。

### 32. 如何用 tc 模拟弱网？

可以使用 `tc netem`：

```bash
sudo tc qdisc add dev ens33 root netem loss 5%
sudo tc qdisc change dev ens33 root netem delay 100ms loss 2%
sudo tc qdisc del dev ens33 root
```

通过模拟丢包、延迟和抖动，可以验证监控指标是否随弱网变化。

### 33. TCP 重传率和 ICMP 丢包率有什么区别？

ICMP 丢包率是 ping 包的丢失情况，反映探测路径上的 ICMP 可达性。

TCP 重传率来自真实 TCP 连接，和业务流量更相关。

某些网络可能限制 ICMP，但 TCP 正常；也可能 ICMP 正常，但 TCP 连接存在拥塞或重传。

## 六、eBPF 与流量分析

### 34. 项目为什么使用 eBPF？

eBPF 可以在内核路径上低开销采集网络事件。

项目使用 eBPF 统计 TCP/UDP 发送路径上的连接信息，包括：

- 源 IP
- 目的 IP
- 源端口
- 目的端口
- 协议
- 字节数
- 包数
- PID

这可以实现 Top Flow 和异常流量分析。

### 35. 什么是五元组？

五元组包括：

```text
源 IP、目的 IP、源端口、目的端口、协议
```

它通常用来唯一标识一条网络连接或流。

### 36. eBPF 在项目中 attach 到哪些点？

项目中 eBPF 程序采样 TCP/UDP 发送路径，例如：

- TCP 发送路径
- UDP 发送路径

用户态通过 libbpf 加载 BPF 对象、attach 程序并读取 BPF map。

### 37. BPF map 在项目中做什么？

BPF map 用于内核态和用户态共享数据。

内核态 eBPF 程序把连接的字节数、包数和 PID 写入 map；用户态程序周期性读取 map，计算 Bps 和 PPS。

### 38. 如何计算 Bps 和 PPS？

每个周期读取两次计数：

```text
Bps = bytes_delta / interval_seconds
PPS = packets_delta / interval_seconds
```

通过 delta 而不是累计值，可以得到实时速率。

### 39. eBPF 为什么可能失败？

常见原因：

- 内核不支持对应 hook。
- 缺少 root 或 capability。
- libbpf 版本不匹配。
- BTF/vmlinux.h 不存在。
- kprobe 符号在不同内核版本上变化。
- 虚拟机环境权限受限。

### 40. eBPF 失败后项目如何处理？

服务不会直接退出，而是进入降级模式：

- Dashboard 使用 `/sys/class/net/<iface>/statistics` 估算流量。
- 仍然保留 RTT、TCP 重传率、网关探测等能力。
- 页面可以继续输出基本诊断。

这是为了提高落地可用性。

### 41. eBPF 和 `/sys/class/net` 统计有什么区别？

`/sys/class/net` 只能给出网卡级别的总流量。

eBPF 可以做到更细粒度：

- 按连接统计。
- 按 PID 统计。
- 支持 Top Flow。
- 支持异常流量定位。

所以 eBPF 是精细分析能力，`/sys` 是基础兜底能力。

### 42. Top Flow 有什么实际价值？

Top Flow 可以定位哪个连接或进程占用大量流量。

例如：

- 某进程持续上传。
- 某服务连接异常重试。
- 某目标 IP 消耗大量带宽。
- 疑似异常流量。

这对现场排障很有价值。

## 七、D-Bus 与客户端库

### 43. 为什么项目使用 D-Bus？

D-Bus 适合 Linux 本地进程间通信。

项目使用 D-Bus 的原因：

- 服务端常驻运行。
- 多个客户端可以查询状态。
- 支持信号通知。
- 适合桌面/终端/本地 agent 场景。

### 44. 项目中 D-Bus 暴露了哪些能力？

主要包括：

- 获取接口列表。
- Ping 指定主机。
- 网络变化信号。
- 接口变化信号。
- 当前上网方式变化信号。
- 网络质量变化信号。

### 45. D-Bus method 和 signal 有什么区别？

method 是请求-响应模式，客户端主动调用服务端。

signal 是发布-订阅模式，服务端主动广播事件，客户端监听。

例如：

- `Ping(host)` 是 method。
- `NetworkQualityChanged` 是 signal。

### 46. 为什么还要封装 `libweaknet.so`？

直接写 D-Bus 调用比较繁琐。

封装动态库后，上层业务只需要调用 C API：

```c
weaknet_init();
weaknet_get_interfaces(...);
weaknet_ping_host(...);
weaknet_subscribe_event(...);
```

这样可以降低集成成本，也便于其他语言通过 FFI 调用。

### 47. 事件订阅在业务侧有什么价值？

业务应用可以根据网络变化做策略调整：

- 弱网时降低码率。
- 网卡切换时重新建连。
- 网络质量变差时提示用户。
- 上报诊断信息到日志系统。

## 八、Web Dashboard 与 SQLite

### 48. Dashboard 为什么用 Python 实现？

Python 开发 Web API、SQLite、日志解析和规则诊断效率较高。

对于短期 MVP，使用 Python 标准库 HTTP Server 可以减少依赖，适合边缘设备部署。

### 49. Dashboard 为什么需要 SQLite？

SQLite 用来缓存结构化指标和事件。

有了 SQLite 后可以：

- 查询最近 N 分钟指标。
- 画趋势图。
- 按网卡过滤。
- 生成诊断报告。
- 为 AI 分析提供结构化数据。

### 50. Dashboard 如何做到实时刷新？

Dashboard 通过两种机制：

- SSE 推送状态。
- 前端定时轮询 `/api/status`、`/api/metrics`、`/api/events`。

后来为了让趋势更实时，把采样周期和前端刷新周期调整到约 1 秒。

### 51. Dashboard 为什么支持点击网卡切换？

多网卡设备上，不同接口的网络质量不同。

点击切换后：

- 顶部指标展示该网卡。
- Trend 按该网卡过滤。
- Events 按该网卡过滤。
- Diagnosis 按该网卡生成。

这样排障时可以区分有线、Wi-Fi、虚拟网卡等不同链路。

### 52. Dashboard 的数据兜底逻辑是什么？

当 C++ 服务端日志或 eBPF 不完整时，Dashboard 直接读取系统数据：

- `/proc/net/route` 获取默认网关。
- `/proc/net/snmp` 计算 TCP 重传率。
- `/sys/class/net/<iface>/statistics` 计算流量。
- 自己 ping `223.5.5.5` 获取 RTT。

这样在 VMware 或权限受限环境下仍能输出可用指标。

### 53. 为什么趋势图有时看起来变化不明显？

可能原因：

- 当前流量接近 0。
- TCP 重传率一直是 0。
- RTT 变化范围小。
- 采样数据很多，但数值平稳。

这不是没有刷新，而是指标本身稳定。

### 54. 为什么 RSSI 显示 N/A？

如果当前网卡不是 Wi-Fi，例如 VMware 的 `ens33`，没有 RSSI 这个指标。

所以显示 `N/A` 是正确行为。

## 九、AI/RAG 辅助分析

### 55. AI 在这个项目中扮演什么角色？

AI 不负责底层采集，而是负责解释和辅助诊断。

底层指标由 C++ 服务和 Dashboard 结构化采集；AI 基于这些数据输出：

- 当前网络状态。
- 异常指标。
- 可能原因。
- 处理建议。
- 证据依据。

### 56. 为什么不能让 AI 直接判断网络状态？

AI 不能替代可靠的底层采集。

如果输入数据不准确，AI 输出也不可靠。因此项目先通过系统能力获取结构化指标，再交给 AI 做解释和建议。

### 57. RAG 在项目中的作用是什么？

RAG 用于把网络诊断知识库和实时指标结合。

流程是：

```text
指标摘要 -> 检索相关知识 -> 拼接上下文 -> AI 生成诊断报告
```

这样可以减少模型幻觉，使回答更贴近网络诊断场景。

### 58. 为什么要构建私有知识库？

网络诊断有很多项目内约定和经验阈值，例如：

- RTT 阈值。
- TCP 重传率判断方式。
- RSSI 解释。
- eBPF 降级说明。
- VMware 环境特征。

这些内容通用模型未必知道，私有知识库可以增强回答的准确性。

### 59. AI 分析为什么应该基于 SQLite，而不是网页？

网页是展示层，不适合作为 AI 输入。

SQLite 中的数据更结构化：

- 指标名称明确。
- 时间戳明确。
- 网卡维度明确。
- 可以按时间窗口查询。

因此更适合生成可靠诊断。

### 60. AI 诊断报告应该包含哪些内容？

建议包含：

- 分析时间窗口。
- 当前活跃网卡。
- 网关 RTT。
- 公网 RTT。
- TCP 重传率。
- RSSI 状态。
- Traffic 状态。
- 发现的问题。
- 可能原因。
- 建议操作。
- 数据依据。

## 十、工程落地与优化

### 61. 当前项目还有哪些可以优化？

- 增加鉴权。

### 62. 为什么质量评分可能一直是 POOR？

因为评分逻辑可能把无效值也算进去了。

例如：

- 有线网卡 RSSI 为 `-1000`。
- RTT 采集失败为 `-1`。
- Traffic 为 0。

如果这些无效值直接参与评分，就会导致误判。

优化方法是只对有效指标加权，指标缺失时重新归一化权重。

### 63. 如何设计更合理的质量评分？

可以采用动态权重：

```text
有效指标参与评分
无效指标不扣分
剩余权重重新归一化
```

例如有线网卡没有 RSSI，就只用 RTT、TCP 重传率、Traffic 计算。

### 64. 如何区分本机问题、网关问题和公网问题？

可以通过分层探测：

- 本机接口状态异常：网卡或驱动问题。
- 网关 RTT 异常：局域网或接入链路问题。
- 网关正常，公网 RTT 异常：上游链路或外网拥塞。
- TCP 重传高：链路丢包、拥塞或服务端问题。
- RSSI 弱：Wi-Fi 信号问题。

- systemd 管理。

监听 `0.0.0.0` 会让局域网其他机器访问 Dashboard。

## 十一、场景题

### 71. 页面显示 RTT 正常，但 Network Quality 是 POOR，怎么排查？

先看质量评分使用的数据源。

可能 RTT 是 Dashboard 兜底 ping 得到的，但 C++ 服务端质量评分仍然使用旧的 `RTT=-1` 或 `RSSI=-1000`。

排查步骤：

1. 查看 `/api/status`。
2. 查看服务端日志中的 RTT。
3. 查看 RSSI 是否为 N/A。
4. 查看 `network_quality_assessor.cpp` 是否把无效值计入评分。

修复方向是质量评分改为动态权重。

### 72. Traffic 一直为 0，怎么排查？

排查步骤：

1. 确认当前活跃网卡，如 `ens33`。
2. 查看服务端日志是否绑定到正确网卡。
3. 检查 eBPF 是否 attach 成功。
4. 查看 `/sys/class/net/ens33/statistics` 计数是否变化。
5. 用 `ping` 或 `curl` 制造流量。

如果 eBPF 不可用，Dashboard 应使用 sysfs 兜底。

### 73. TCP Loss 出现很高尖峰，怎么判断是否真实？

不能只看单个 max。

应检查：

- 样本数量是否足够。
- OutSegs 分母是否过小。
- 是否连续多个周期高。
- 平均值和 P95 是否也高。
- 是否和 RTT 升高、Traffic 增大同时发生。

偶发尖峰可能只是短窗口统计误差。

### 74. 网关 RTT 正常但公网 RTT 高，说明什么？

说明本机到局域网出口正常，但上游链路可能存在问题，例如：

- 宽带出口拥塞。
- NAT/路由器上游异常。
- 公网 DNS 目标路径拥塞。
- 运营商链路问题。

### 75. RSSI 弱但 TCP Loss 不高，怎么解释？

可能当前无线信号较弱，但尚未导致明显传输层重传。

需要继续观察：

- RTT 是否抖动。
- RSSI 是否持续下降。
- Traffic 上升时是否出现重传。
- 是否有漫游或 AP 切换。

### 76. 如何定位某个应用网络异常？

可以结合：

- 应用 PID。
- eBPF Top Flow。
- 目标服务器域名/IP。
- 该连接的 Bps/PPS。
- RTT 和 TCP 重传率。

未来可以扩展接口：传入 PID 和服务器域名，返回该应用链路上的网络指标。

### 77. 为什么要把诊断报告做成中文？

因为目标用户可能是现场运维、测试或业务开发。中文报告能降低理解成本，使网络连通性问题更容易被定位和流转到 bug 单中。

### 78. 如何避免 AI 诊断胡说？

关键是限制输入和输出：

- 输入必须来自结构化指标。
- 提供明确时间窗口。
- 提供知识库上下文。
- 提示词固定输出格式。
- 报告必须引用数据依据。
- 无数据时明确说明无法判断。

### 79. 这个项目和普通监控系统有什么不同？

普通监控系统更偏中心化和服务端指标。

本项目更偏端侧：

- 关注当前设备自身网络链路。
- 能识别当前上网网卡。
- 能看到网关、RSSI、TCP 重传和本机流量。
- 可在没有中心平台时本地运行。

### 80. 如何把这个项目扩展为企业级平台？

可以增加：

- 端侧 agent 安装包。
- 统一配置中心。
- 多设备指标上报。
- 中心化 Dashboard。
- 告警规则。
- 设备分组。
- 远程诊断。
- AI 批量分析报告。

## 十三、源码级深挖补充问题

### 91. 面试官问服务端启动后具体创建了哪些线程，怎么回答？

可以回答：

服务端入口是 `main.cpp`，只负责调用 `weaknet_dbus::start_server()`。在 `start_server()` 里先初始化日志和 D-Bus，然后初始化 `WeakNetMgr` 的网卡列表，之后启动多个后台线程：

- 网卡列表监控线程：周期性扫描可上网网卡，发现增删时发 `InterfaceChanged`。
- 当前上联网卡监控线程：基于默认路由和 netlink 状态更新 `usingNow`，变化时发 `ConnectionModeChanged`。
- RTT 监控线程：周期性对目标地址如 `223.5.5.5` 做 ping，更新 RTT 和链路质量。
- RSSI 监控线程：针对 Wi-Fi 接口读取信号强度。
- TCP loss 监控线程：采样 TCP 统计并计算重传率。
- Traffic 分析线程：启动 eBPF 或降级流量分析。
- Network Quality 线程：综合各指标计算质量等级并推送质量事件。

主线程最后进入 `Looper::run()`，负责 D-Bus 消息循环。

### 92. 为什么服务端采用“多线程周期采集”而不是单线程事件驱动？

可以回答：

因为这些指标的数据源不同，有些是事件型，有些只能周期采样。比如路由和网卡变化可以通过 netlink 监听，但 RTT、TCP 重传差值、RSSI、流量速率都需要固定时间窗口计算。多线程能让不同指标按各自周期独立更新，避免某一个探测阻塞整个服务。

缺点是共享状态需要加锁，指标时间点不完全一致，所以项目里通过 `WeakNetMgr` 集中维护 `NetInfo`，并在 Dashboard 侧用时间窗口聚合，降低时序不一致带来的误判。

### 93. `WeakNetMgr` 在项目里是什么角色？

可以回答：

`WeakNetMgr` 是服务端的状态聚合中心。底层采集器只负责拿数据，最后都会把结果更新到 `WeakNetMgr` 管理的 `NetInfo` 列表里。`NetInfo` 保存网卡名、是否当前使用、RTT、RSSI、TCP loss、Traffic、质量等级等字段。

这样做的好处是上层质量评估、日志输出、事件判断都可以从统一状态读取，而不是每个模块各自维护一份状态。

### 94. `NetInfo` 里为什么要用 `RTT=-1`、`RSSI=-1000` 这种特殊值？

可以回答：

这是为了区分“采集失败/不适用”和正常数值。比如 RTT 采集失败时不能填 0，因为 0ms 会被误认为网络极好；非 Wi-Fi 网卡没有 RSSI，如果填真实范围内的数值也会误导评分。所以用明显不在正常范围内的哨兵值表示不可用。

但这也带来一个要求：所有评分和展示逻辑必须识别这些哨兵值，不能直接把它们当成真实指标参与计算。

### 95. 当前项目里质量评分有什么不足？

可以回答：

当前 `NetworkQualityAssessor` 采用固定权重：RTT 30%、TCP loss 30%、RSSI 20%、Traffic 20%。这个方案简单直观，但问题是当某个指标不适用时，比如有线网卡没有 RSSI，固定权重仍可能影响总分。

更合理的方案是动态权重：只让有效指标参与评分，缺失指标不扣分，剩余权重重新归一化。例如有线网卡可只使用 RTT、TCP loss 和 Traffic；Wi-Fi 网卡再额外加入 RSSI。

### 96. 面试官问“你这个项目真的用到了 netlink 吗”，怎么回答？

可以回答：

用到了，主要在当前上联网卡识别模块。`UsingInterfaceManager` 创建 `NETLINK_ROUTE` socket，订阅 `RTMGRP_LINK`、`RTMGRP_IPV4_ROUTE`、`RTMGRP_IPV6_ROUTE`，监听链路状态和路由变化。

启动时会 dump 当前 link 和 route，运行中处理 `RTM_NEWLINK`、`RTM_DELLINK`、`RTM_NEWROUTE`、`RTM_DELROUTE`，维护 UP 网卡集合、IPv4 默认路由集合和 IPv6 默认路由集合，最后选出当前默认出口网卡。

### 97. netlink 判断当前上联网卡的逻辑是什么？

可以回答：

逻辑大致是：

1. 通过 `RTM_GETLINK` 获取网卡 ifindex 和名称，并过滤 loopback。
2. 记录处于 UP 状态的网卡。
3. 通过 `RTM_GETROUTE` 获取默认路由，默认路由的目标前缀长度是 0。
4. 从默认路由里取 `RTA_OIF` 得到出口 ifindex。
5. 优先选择既是默认路由出口、又处于 UP 状态的网卡。
6. 同时记录 IPv4/IPv6 默认路由标志，便于说明当前上网方式。

### 98. 为什么不能只用 `/proc/net/route` 判断当前网卡？

可以回答：

`/proc/net/route` 可以拿到 IPv4 默认路由，但它更偏快照查询，不方便实时监听变化，也不覆盖所有 IPv6 场景。netlink 不仅能查询初始状态，还能订阅路由和链路变化事件，适合常驻服务实时更新当前上联网卡。

Dashboard 用 `/proc/net/route` 做兜底是合理的，但服务端核心识别逻辑更适合用 netlink。

### 99. D-Bus 服务注册了哪些方法和信号？

可以回答：

项目里的 D-Bus 基本标识是：

- Bus name：`com.example.WeakNet`
- Object path：`/com/example/WeakNet`
- Interface：`com.example.WeakNet`

方法包括：

- `Get`：示例性健康检查返回。
- `ListInterfaces` / `GetInterfaces`：返回接口列表。
- `Ping`：对指定 hostname 通过当前上联网卡发起 ping。

信号包括：

- `Changed`：通用状态变化。
- `InterfaceChanged`：网卡变化。
- `ConnectionModeChanged`：当前上网方式变化。
- `NetworkQualityChanged`：网络质量变化。

### 100. 当前 `Get` 方法为什么不是一个完整状态接口？

可以回答：

当前 `Get` 方法更像连通性和示例接口，返回的是固定字符串，并把结果序列化到文件里。完整状态目前主要通过日志和 Dashboard 解析来获得。

这也是项目后续最优先优化点之一：新增 `GetStatusJson` 或类似 D-Bus 方法，把 `WeakNetMgr` 当前维护的全部 `NetInfo` 状态结构化返回，避免 Dashboard 依赖日志格式。

### 101. 代码里 `ListInterfaces` 可能有什么问题？

可以回答：

当前 `DbusService::handleListInterfaces()` 读取的是 `ctx_->iface_list`，但服务端主要更新的是 `WeakNetMgr` 内部的 `current_interfaces_`。如果 `ctx_->iface_list` 没有同步更新，`ListInterfaces` 可能拿不到最新列表。

修复思路是让 D-Bus 接口直接调用 `ctx_->weak_mgr->getCurrentInterfaces()`，再转换成接口名数组返回，保证和采集状态一致。

### 102. D-Bus Signal 和 Method Call 的区别是什么？项目里为什么两者都用？

可以回答：

Method Call 是请求-响应模式，适合主动查询，比如 `Ping`、`GetInterfaces`；Signal 是发布订阅模式，适合异步通知，比如网卡变化、网络质量变化。

项目里两者结合：上层应用需要当前状态时主动调用方法；网络变化发生时服务端通过 Signal 主动推送，避免客户端频繁轮询。

### 103. 为什么客户端还要封装成 `libweaknet.so`？

可以回答：

直接使用 libdbus 写客户端代码比较繁琐，需要处理连接、消息构造、参数解析、订阅规则和错误处理。封装成 `libweaknet.so` 后，上层只需要调用 C API，比如 `weaknet_init()`、`weaknet_ping_host()`、`weaknet_check_events()`。

另外 C ABI 稳定，方便其他语言通过 FFI 集成，比直接暴露 C++ 类更通用。

### 104. 客户端的非阻塞事件检查怎么实现？

可以回答：

客户端先通过 D-Bus match rule 订阅指定 signal，然后调用 `dbus_connection_read_write(conn, 0)` 做非阻塞轮询，再用 `dbus_connection_pop_message()` 取消息。如果没有消息就立即返回 false；如果有目标 signal，就解析出 event type、message、counter 等信息。

这样业务线程可以把事件检查集成到自己的循环里，不会因为等待 D-Bus 消息而阻塞。

### 105. 为什么事件系统还要有 `NetworkEventManager`，不直接在各线程里发 D-Bus Signal？

可以回答：

直接发 Signal 会让采集逻辑和通信逻辑耦合。`NetworkEventManager` 做了统一事件抽象，包含事件类型、消息、来源、优先级、详情字段，并支持本地 callback 和 D-Bus signal 两种出口。

这样后续如果增加 WebHook、持久化事件、告警规则，不需要改每个采集线程，只需要扩展事件管理器。

### 106. `NetworkQualityChanged` 为什么要带 details？

可以回答：

普通事件只需要一个 message，但网络质量事件需要说明评分依据，比如接口名、RTT、TCP loss、RSSI、Traffic、issues 等。details 用 JSON 字符串表达这些结构化信息，客户端或 Dashboard 可以进一步解析，而不仅仅显示一段文本。

### 107. eBPF 程序挂载在哪些内核函数上？

可以回答：

当前 eBPF 程序主要挂载两个 kprobe：

- `ip_queue_xmit`：统计 TCP 发送路径。
- `udp_sendmsg`：统计 UDP 发送路径。

TCP 侧可以从 `sock` 和 `sk_buff` 里提取 IPv4 五元组、包长度、网卡 ifindex；UDP 侧从 `sock` 和函数参数长度统计 UDP 发送流量。

### 108. eBPF map 为什么使用 LRU Hash Map？

可以回答：

连接级统计的 key 是五元组，活跃连接数量可能很多。如果普通 Hash Map 不清理，长时间运行会积累旧连接，占用内核内存。LRU Hash Map 在超过容量时会自动淘汰较久未使用的条目，适合这种流量窗口统计场景。

项目里 `current_sec` 最大 entries 是 65536，能覆盖较多活跃连接，同时避免无限增长。

### 109. eBPF 统计的 key 和 value 分别是什么？

可以回答：

key 是连接五元组：

- 源 IP
- 目的 IP
- 源端口
- 目的端口
- 协议号，TCP 是 6，UDP 是 17

value 是：

- bytes
- packets
- pid

通过这个结构可以计算每条连接的 Bps、PPS，并定位到进程维度。

### 110. eBPF 为什么当前只统计 IPv4？

可以回答：

当前 `fill_key_from_sock()` 判断 `skc_family`，只处理 `AF_INET`。这是为了先降低实现复杂度，因为 IPv4 的地址和端口提取比较直接。IPv6 需要处理 128 位地址、不同字段和展示格式，key 结构也要扩展。

面试时可以说明：这是当前版本的边界，不是理论限制；后续可以扩展 IPv6 key 结构和解析逻辑。

### 111. eBPF 的 UDP 网卡过滤为什么比 TCP 弱？

可以回答：

TCP kprobe 在 `ip_queue_xmit` 上可以拿到 `sk_buff`，进而读取 `skb->dev->ifindex` 做网卡过滤；UDP 挂在 `udp_sendmsg`，这个位置没有直接拿到 skb，代码里为了兼容性没有强行读取旧内核字段，所以 UDP 当前统计总量，不做严格接口过滤。

这属于准确性和兼容性的取舍。如果后续要增强，可以选择更靠近网络发送路径的位置统一处理，或者用 tracepoint/tc hook 获取更稳定的 skb 上下文。

### 112. Top Flow 是怎么计算出来的？

可以回答：

用户态会读取 eBPF map 做两次快照。第一次记录每条 flow 的 bytes 和 packets，然后 sleep 一个采样窗口，比如 5 秒；第二次再读 map，计算 delta bytes 和 delta packets，得到：

```text
bps = delta_bytes / interval
pps = delta_packets / interval
```

最后按 bps 排序，取前 N 个作为 Top Flow。

### 113. 流量异常检测是怎么做的？

可以回答：

当前主要有三类规则：

- burst：当前流量超过阈值，并且超过历史均值的一定倍数。
- suspicious：超过可疑流量阈值。
- high_volume：超过高流量阈值。

每条 flow 会维护历史 Bps/PPS 序列，基于历史均值和当前值判断是否突发。当前规则偏启发式，适合做第一版告警，后续可以结合业务白名单、端口画像和进程名增强准确性。

### 114. eBPF 不可用时系统会怎样？

可以回答：

eBPF 初始化失败时，TrafficAnalyzer 会记录失败并进入 degraded mode，不让整个服务退出。这样虽然拿不到连接级 Top Flow 和 PID 粒度，但 Dashboard 仍可以通过 `/sys/class/net/<iface>/statistics` 计算网卡总流量，通过 `/proc/net/snmp` 计算 TCP 重传。

这个降级设计保证项目在权限不足、内核不支持、虚拟机环境下仍然可用。

### 115. 为什么 Dashboard 没有直接调用 D-Bus？

可以回答：

当前 Dashboard 主要服务两个目标：展示实时指标和做短期趋势分析。它通过 tail 服务端运行日志获得历史序列，同时读取 `/proc`、`/sys` 做兜底采样，再写入 SQLite。这样即使 D-Bus 状态接口还不完整，也能实现趋势图、事件列表和诊断报告。

缺点是和日志格式耦合，所以后续会把 Dashboard 的主数据源切到 D-Bus 结构化接口，日志解析只作为兼容或审计来源。

### 116. Dashboard 为什么同时使用 SSE 和轮询？

可以回答：

SSE 用来推送状态变化，延迟低，适合实时状态卡片；轮询用于定期刷新 metrics/events/report 这类查询数据，逻辑简单且能处理 SSE 断连的情况。

两者组合可以提高可用性：SSE 正常时页面更实时，SSE 失败时轮询仍然能更新数据。

### 117. Dashboard 的 SQLite 表如何设计？

可以回答：

主要有两类表：

- `metrics`：存储时间戳、interface、RTT、quality、using_now、RSSI、TCP loss、traffic、flows、pps、source、raw_line 等。
- `events`：存储时间戳、type、severity、interface、message、details、raw_line 等。

同时对 `ts` 和 `interface` 建索引，方便按时间窗口和网卡过滤查询。

### 118. Dashboard 如何避免重复写入同一条日志？

可以回答：

解析每条日志时会根据记录类型、原始行和解析出的 data 生成 hash，写入 SQLite 时把这个 hash 作为唯一键。这样即使 LogTailer 回填最近一段日志，已经处理过的记录也不会重复插入。

### 119. 为什么 Dashboard 要 backfill 最近 512KB 日志？

可以回答：

Dashboard 启动时如果只从文件末尾开始读，页面初始会没有历史数据。backfill 最近一段日志可以让页面启动后立即展示近期状态和趋势。

限制在 512KB 是为了避免日志很大时启动解析过慢，也避免边缘设备资源占用过高。

### 120. `report` 接口的诊断逻辑是什么？

可以回答：

`/api/report` 会取指定时间窗口内的 metrics 和 events，然后对 RTT、TCP loss、RSSI、Traffic 做清洗和统计。比如计算平均值、P95、最大流量等，再根据阈值生成 issues 和 suggestions。

它是规则诊断，不依赖外部 AI，所以在没有网络或没有 API key 的情况下也能生成基础中文诊断报告。

### 121. RAG 模块的本地向量是怎么做的？

可以回答：

本地 RAG 先从 `network_knowledge_base.py` 读取网络知识库，再切分成文档块。嵌入层使用一个简单的本地 Hash/词频向量，把文本映射到固定维度向量，然后用 FAISS 做相似度检索。

如果配置了 DashScope API，就把检索到的知识上下文和当前指标一起交给模型生成报告；如果没有 API，就走本地规则兜底分析。

### 122. 为什么本地 embedding 不用大型 embedding 模型？

可以回答：

这是为了降低部署门槛。项目目标是 Linux 端和边缘环境，可能没有 GPU，也不一定能访问外网。本地 Hash/词频向量虽然语义能力有限，但对小型专业知识库和关键词明确的网络诊断场景足够做基础检索。

如果做生产版本，可以替换成更好的 embedding 模型或统一调用云端 embedding 服务。

### 123. RAG 如何减少 AI 胡说？

可以回答：

主要靠三点：

1. 输入必须来自结构化指标，而不是让模型自由猜。
2. 检索网络知识库作为上下文，比如 RTT、TCP loss、RSSI 的判断规则。
3. 输出要求包含依据，数据不足时明确说无法判断。

AI 在项目里定位是解释器和报告生成器，不是底层采集和事实来源。

### 124. 面试官问“这个项目和 Prometheus node_exporter 有什么区别”，怎么回答？

可以回答：

node_exporter 更偏通用主机指标采集，适合中心化监控；这个项目更偏端侧网络诊断 agent，关注当前设备的上网链路质量。它不仅采集网卡 counters，还识别当前上联网卡、网关、RTT、RSSI、TCP 重传、Top Flow，并通过 D-Bus 给本机应用推送网络事件。

所以它不是替代 Prometheus，而是补充端侧弱网诊断能力。企业化后也可以把本项目的结果再上报到 Prometheus。

### 125. 面试官问“为什么不用 tcpdump 直接抓包”，怎么回答？

可以回答：

tcpdump 适合人工排查，但不适合作为长期常驻服务的默认方案。它输出量大、解析成本高、对权限和性能影响更明显，也不方便直接聚合成每个连接的统计指标。

eBPF 更适合在内核路径上做低开销聚合，只把统计结果带到用户态，避免大量原始包复制。

### 126. 面试官问“为什么不用 HTTP API 作为 C++ 服务端对外接口”，怎么回答？

可以回答：

C++ 服务端定位是本机 agent，D-Bus 是 Linux 本地进程间通信的常用机制，天然支持本机服务发现、方法调用和信号通知。HTTP 更适合 Dashboard 面向浏览器。

如果未来要做远程管理，可以在上层增加 HTTP/gRPC 网关，而不是让底层 agent 直接暴露网络端口。

### 127. 当前服务端线程生命周期有什么可以改进的地方？

可以回答：

当前有些线程是 `detach`，有些线程保存在 `ServerContext` 中，整体能运行，但停止和回收不够统一。更好的方式是所有后台线程都由统一生命周期管理：启动后保存句柄，退出时设置 `running=false`，再逐个 join。

这样可以减少服务关闭时的资源泄漏风险，也便于后续做 systemd stop、reload 或单元测试。

### 128. 如果 D-Bus 发送信号失败怎么办？

可以回答：

当前代码主要记录发送结果，失败不会影响采集线程继续运行。因为采集服务的核心职责是持续维护状态，Signal 是通知通道，不应因为某个客户端不可用导致服务停止。

更完善的做法是增加失败计数、日志告警和最近事件缓存。客户端重连后可以通过状态查询接口补偿，而不是只依赖实时 Signal。

### 129. 如何保证采集线程不会被慢操作拖垮？

可以回答：

项目里对一些可能阻塞的操作设置了超时或异步处理，比如 ping 有 timeout，D-Bus signal 发送放到短线程里避免阻塞采集循环。采样周期也不会太短，常见是 10 秒或 15 秒。

后续可以继续优化为线程池或事件队列，避免频繁创建临时线程，并给每个采集器增加耗时统计。

### 130. 如果网卡频繁切换，会发生什么？

可以回答：

netlink 和周期扫描都会感知接口变化。`UsingInterfaceManager` 更新当前默认出口，`WeakNetMgr` 更新每个 `NetInfo` 的 `usingNow`，变化时触发 `ConnectionModeChanged`。流量分析线程发现 active iface 变化后会停止旧接口分析，并重新绑定到新接口。

需要注意的是频繁切换会造成短时间内指标不完整，比如新接口刚启用时 RTT、TCP loss、Traffic 还没形成完整窗口。诊断时应允许一个短暂冷启动窗口。

### 131. 如何解释“异构指标时间对齐”这个难点？

可以回答：

RTT 是主动探测，TCP loss 是两个采样点的差值，Traffic 是 eBPF 或 sysfs 速率窗口，RSSI 来自 Wi-Fi 控制接口，它们不是同一时刻产生的。如果直接拿最新值做判断，可能 RTT 是 10 秒前的，TCP loss 是当前窗口的，RSSI 又是另一个时间点的。

项目通过统一写入状态、记录日志时间戳、Dashboard 按时间窗口聚合来缓解。生产级方案还可以给每个指标加采样时间和有效期，评分时只使用同一时间窗口内的新鲜数据。

### 132. 如何解释“弱网误判控制”这个难点？

可以回答：

弱网不是单个指标超过阈值就能确定。RTT 高可能是公网目标波动，TCP loss 短窗口可能因为分母太小而尖峰，RSSI 对有线网卡无意义，Traffic 为 0 可能只是没有业务流量。

所以要做有效值过滤、最小样本门槛、多指标交叉验证和时间窗口聚合。比如 RTT 高并且 TCP loss 高，可信度更高；只有一次 TCP loss 尖峰，则更倾向于观察而不是直接告警。

### 133. 如何解释“底层能力兼容”这个难点？

可以回答：

底层能力依赖环境：eBPF 要内核、BTF、权限和 libbpf；RSSI 要真实 Wi-Fi 网卡和 wpa_supplicant；ICMP 可能需要权限；虚拟机网卡没有无线信号。项目必须承认这些能力不是总能拿到。

因此设计上要分层降级：eBPF 不可用就用 sysfs 总流量；RSSI 不可用显示 N/A；服务端 RTT 不可用时 Dashboard 做独立探测；AI 不可用时规则诊断仍能工作。

### 134. 如果面试官问“你怎么验证这个项目有效”，怎么回答？

可以回答：

我会从三类验证入手：

1. 功能验证：启动服务端和 test-client，验证 D-Bus 方法调用、Ping、事件订阅、非阻塞事件检查。
2. 指标验证：用 `ping`、`tc netem`、`curl`、切换网卡等方式制造延迟、丢包、流量和接口变化，检查日志和 Dashboard 是否反映变化。
3. 降级验证：在虚拟机、有线网卡、无 eBPF 权限环境下运行，确认系统不会崩溃，Dashboard 仍能显示基础指标。

### 135. 如果面试官问“你项目里最有技术含量的点是什么”，怎么回答？

可以回答：

不是单独某个技术点，而是把 Linux 网络采集、D-Bus 本地 IPC、eBPF 连接级流量统计、Web 可视化、SQLite 时间窗口查询和 RAG 诊断串成闭环。尤其是底层指标存在缺失和误差时，仍能通过降级、过滤和多指标融合输出可解释诊断，这是最体现工程能力的部分。

### 136. 如果面试官追问“你会怎么把日志解析替换掉”，怎么回答？

可以回答：

我会在 C++ 服务端新增结构化状态接口，例如：

```text
GetStatusJson() -> JSON
GetMetricsSnapshot() -> array of NetInfo
GetRecentEvents(limit) -> events
```

Dashboard 优先调用这些接口，拿到结构化字段后写入 SQLite。日志只保留给人工排查和审计。这样可以减少正则解析脆弱性，也能避免日志格式变化导致页面失效。

### 137. 如果面试官问“这个项目有没有安全风险”，怎么回答？

可以回答：

有几个风险点：

- Dashboard 如果监听 `0.0.0.0` 且无鉴权，会暴露本机网络状态。
- eBPF 和 ICMP 可能需要较高权限，需要最小权限运行。
- RAG/API key 不能硬编码进仓库。
- D-Bus 接口需要考虑调用方权限，避免任意本机进程滥用。

生产化需要加鉴权、配置化密钥、systemd 权限限制、capability 最小化和 D-Bus policy。

### 138. 如果面试官问“为什么项目里有日志乱码”，怎么回答？

可以回答：

这是编码显示问题，不一定是业务逻辑问题。项目里很多中文注释和日志是 UTF-8，但在 Windows PowerShell 默认编码或某些终端环境下会按错误编码显示，所以看起来是乱码。

工程上应该统一源码和日志为 UTF-8，运行脚本和文档说明编码要求；如果面向生产，日志建议尽量使用英文 key + 结构化字段，中文只放在展示层。

### 139. 如果面试官问“当前项目最容易被质疑真实性的地方是什么”，怎么回答？

可以回答：

最容易被追问的是 eBPF、netlink、D-Bus 是否真的落地。回答时不要泛泛说“用了”，而要讲清楚：

- netlink：`NETLINK_ROUTE` 监听 link/route，识别默认出口网卡。
- D-Bus：注册 `com.example.WeakNet`，提供 Method 和 Signal。
- eBPF：kprobe `ip_queue_xmit`、`udp_sendmsg`，LRU Hash Map 统计五元组 bytes/packets/pid。
- Dashboard：tail runtime.log，解析后写 SQLite，提供 `/api/status`、`/api/metrics`、`/api/events`、`/api/report`。

能说出这些实现细节，可信度会高很多。

### 140. 如果面试官让你画系统数据流，你怎么口述？

可以回答：

数据从 Linux 内核和系统文件来：netlink 提供路由和网卡变化，ICMP ping 提供 RTT，`/proc/net/snmp` 提供 TCP 统计，wpa_supplicant 提供 RSSI，eBPF 提供连接级流量。C++ 服务端把这些数据聚合到 `NetInfo`，通过 D-Bus 方法和 Signal 对外提供，同时写运行日志。Dashboard 解析日志并读系统计数器兜底，写入 SQLite 后通过 Web API 和 SSE 展示。AI/RAG 再基于结构化日志或指标生成诊断建议。
